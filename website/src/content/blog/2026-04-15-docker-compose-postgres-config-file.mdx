I am Philip, an engineer working at Distr, which helps software and AI companies distribute their applications to self-managed environments.
Our Open Source Software Distribution platform is available on GitHub ([`github.com/distr-sh/distr`](https://github.com/distr-sh/distr)).

A customer came to us last week with a very practical question: they ship a PostgreSQL container as part of their product, and not all the settings they care about can be driven by `POSTGRES_*` environment variables.
The upstream Postgres image documents a `postgresql.conf` mount as the recommended path, but when your deployment is a single Docker Compose file handed to a customer, "just mount a file from the host" is more friction than it sounds.

Docker Compose has a native answer for this. No custom image, no tarball alongside the Compose file, no pre-install script: a short `configs:` block at the bottom of the Compose file is enough.

## Why POSTGRES\_\* environment variables don't cover postgresql.conf

The official Postgres image exposes a handful of environment variables: `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB`, `POSTGRES_INITDB_ARGS`, and a few related ones. Those are enough to bootstrap a database, but they deliberately stop short of the server tuning knobs.

Settings like `shared_buffers`, `max_connections`, `work_mem`, `effective_cache_size`, `wal_level`, `shared_preload_libraries`, or locale-related options like `lc_messages` are not exposed through environment variables at all. The upstream image documentation explicitly points you at two alternatives: override the startup command with `-c key=value` flags, or mount a `postgresql.conf` file and start the server with `config_file=...`.

`-c` flags work for one or two tweaks, but the moment you have more than a handful, you end up with a wall of arguments that nobody wants to review in a pull request. A real `postgresql.conf` is the right tool, and it is also what most DBAs will expect to see when they onboard your application.

This works fine on a developer laptop where the Compose file and the config live side by side in a git checkout. It stops working cleanly the moment you distribute the deployment as a single artifact.

A few concrete problems show up:

- **Two files instead of one.** Your customer now has to place `postgresql.conf` on the host at exactly the right path before `docker compose up` will succeed. If they clone the wrong directory or run Compose from somewhere unexpected, they get a cryptic "bind source path does not exist" error.
- **File ownership and permissions drift.** On some hosts the bind-mounted file ends up owned by a user the Postgres process inside the container cannot read, especially when the container runs as a non-root user.
- **Editability is unclear.** "Editable by the user" was literally the request from our customer. A bind-mounted host file is editable, but nothing in the Compose file signals _which_ file the customer is supposed to edit, or what the defaults were.
- **Shipping updates is harder.** When you roll out a new version of your application and the recommended defaults change, you now have two moving parts to coordinate: the Compose file and the external config file.

The cleaner fix is not a Distr feature at all. It is a Docker Compose feature that has been quietly sitting in the spec for a couple of years.

## Using Docker Compose `configs` with inline `content` for `postgresql.conf`

Docker Compose has a top-level `configs` keyword that was originally carried over from Docker Swarm. Since Compose version 2.23.1 it supports an inline `content:` field, which lets you embed a config file directly in the Compose file and mount it into a service.

We already use this pattern in one of our own open source projects, the [hyprmcp `mcp-gateway`](https://github.com/hyprmcp/mcp-gateway/blob/main/examples/who-am-i/docker-compose.yaml), to inline a Dex OIDC configuration without shipping a second file alongside the Compose file. A few things worth calling out in that file:

- The top-level `configs:` block at the bottom declares `postgresql.conf` with an inline `content:` YAML block. Compose will create the file at container start time from whatever is in that heredoc.
- The service references it with `source:` and `target:`. `target:` is important here because the default mount location for a config is `/<config-name>`, and Postgres expects its config at a specific path passed via `config_file=`. Using `target: /etc/postgresql/postgresql.conf` lines things up.
- The `command:` override tells Postgres to read its config from that path instead of the baked-in defaults. The entrypoint of the official `postgres` image passes any trailing arguments through to the `postgres` binary, so the container still runs `initdb` on first start and honors `POSTGRES_USER` and friends.
- Configs are mounted read-only (mode `0444`) by default, which matches the upstream image's expectations.

<Aside type="note">
  Older versions of Docker Compose do not understand the `content:` field.
  Depending on the exact version you either get a schema error at parse time or
  an empty file inside the container, followed by a Postgres `could not open
  configuration file` failure. Require Compose `>= 2.23.1` in your installation
  instructions, or fall back to an external `file:` source if you cannot.
</Aside>

## How Distr ships this single-file Postgres deployment

The reason this pattern matters for us is that [Distr](https://distr.sh) deployments are a single Compose file handed to an agent running in the customer's environment. The customer does not see a git repository, there is no separate "config directory", and the agent pulls the exact bytes of the Compose file from our control plane.

With the `configs` keyword, everything the customer needs to run the application lives inside that one file: services, networks, volumes, _and_ the Postgres tuning. You still control what the customer is allowed to change directly in Distr, and because the Postgres config is just YAML inside the Compose file, any parameter you want to surface can be templated out through Compose variables:

```yaml
configs:
  postgresql.conf:
    content: |
      listen_addresses = '*'
      max_connections = ${PG_MAX_CONNECTIONS:-200}
      shared_buffers = ${PG_SHARED_BUFFERS:-512MB}
```

The customer can then override `PG_MAX_CONNECTIONS` through the Distr UI or their environment, and Compose will render the value into the config file at container start. This gives you a "shipped defaults, editable when it matters" setup without building a custom Postgres image.

## Three practical notes before you use Compose `configs` in production

I have tripped over each of these at least once while helping customers move to this pattern:

- **Check the minimum Compose version on the target hosts.** Compose `2.23.1` shipped in November 2023 and is bundled with current Docker Desktop and Docker CE. But if your customers still run the legacy Compose v1 Python binary (`docker-compose` with a dash), the `configs` block with inline `content:` is not going to work. Print the expected version in your installer and fail fast if it is lower.
- **Inline content is not a secret store.** The file is plain text inside your Compose file. Do not put database passwords, license keys, or TLS private keys in a `configs` block. For those, use the Compose `secrets` top-level keyword, an external secret manager, or [Distr Secrets](/docs/agents/secrets/), which are stored centrally in Distr and substituted into deployments at deploy time using `{{ .Secrets.KEY }}` template syntax so the actual values never land in your Compose file or deployment logs. `configs` is intended for configuration in the boring sense: tuning parameters, application settings, static templates.
- **Locale and extension assumptions still belong to the image.** `shared_preload_libraries = 'age'` or `lc_messages = 'en_US.UTF-8'` only work if your Postgres image actually ships the `age` extension and the `en_US.UTF-8` locale. Switching to `configs` does not change that. You are still responsible for picking an image that has what your `postgresql.conf` asks for.

## Conclusion

The customer question that started this post was effectively "can Distr mount editable config files for me". Docker Compose already can, and Distr ships whatever valid Compose file you give it. Inline `configs:` with `content:` is the cleanest way I have found to ship a `postgresql.conf`, or any other config file a container needs, without introducing a second artifact or a custom image.

If you are shipping an application with Postgres, Redis, Nginx, Dex, or anything else that expects a real config file, move the file into a `configs:` block. Your Compose file stays self-contained, your customer sees exactly what they are running, and you keep the ability to template the parts they should be able to change.
